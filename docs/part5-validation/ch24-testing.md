

# 第23章 船舶与陆地测试体系
## 台架、海试、全寿命验证方法

> **本章导读**
>
> 测试体系是将设计转化为可靠产品的**最后一道防线**。不完善的测试会导致：
> - 设计缺陷逃逸到现场
> - 故障率高企(MTBF<1000h)
> - 需要大量现场改装与维修
> - 品牌信誉受损
>
> 本章系统阐述从**台架测试→海试验证→全寿命跟踪**的完整体系。
>
> **本章关键输出物**：
> - 陆地台架测试规范(动力计、测功机、耐久性测试)
> - 海试验证计划(包括极限工况仿真)
> - FAT/SAT验收准则(船级社审图标准)
> - 全寿命数据追踪体系(云端实时监控)
> - 故障反馈与设计改进的闭环

### 20.1.4 冷却系统1000h耐久性测试

**测试流程**：

```
Phase 1: 初期运行(0-100h)
  目的：检验系统是否存在明显缺陷
  工况：恒功率100kW，恒定流量
  重点监测：
    • 泵是否能正常启动
    • 流量传感器读数是否稳定
    • 温度是否能控制在±3°C范围内
    • 是否有泄漏(听诊+压力监测)
  
  合格标准：零故障，各参数稳定
  
  失败处理：停机排查，修复后重新启动

Phase 2: 中期运行(100-500h)
  目的：考核系统长期稳定性与可靠性
  工况：变工况(25%-100% 功率循环)
        每4小时切换一次工况点
  
  重点监测：
    • 冷却液质量(定期取样分析)
    • 泵的噪声(连续监听/超声波)
    • 阀门是否泄漏或卡死
    • 电机温度-流量的关联性
    • 压力传感器准确性
  
  合格标准：
    □ 无故障停机
    □ 流量变化<5%
    □ 温度波动<2°C
    □ 液质无异常(无颜色变化/无固体颗粒)

Phase 3: 长期验证(500-1000h)
  目的：验证整体系统是否满足5年寿命
  工况：高温应力(恒温50°C ±2°C)
        连续运行，每周检验一次
  
  加速因子：高温加速电池化学反应
        计算：Arrhenius equation
        1周@50°C ≈ 4周@25°C
  
  重点监测：
    • 液体黏度变化(高温下挥发)
    • 滤芯堵塞趋势
    • 腐蚀/沉积(金属元素分析)
    • 磨损颗粒(铁/铜/铝含量)
  
  合格标准：
    □ 无阶段性故障
    □ 冷却液仍可用(1000h后可继续用至寿命末)
    □ 所有传感器偏差<5%
    □ 泵性能衰减<10%

输出报告：
  ① 1000h运行日志(每天关键参数汇总)
  ② 故障(若有)分析与纠正措施
  ③ 冷却液定期取样报告(3份：100h/500h/1000h)
  ④ 综合可靠性评估
  ⑤ 改进建议与设计优化
```

## 20.2 海试验证计划

### 20.2.1 FAT(工厂验收测试) vs SAT(现场验收测试)

```
┌────────────────────────────────────────────────────┐
│           FAT vs SAT 对标表                         │
├────────────────────────────────────────────────────┤

项目           │  FAT(工厂)              │  SAT(现场)
───────────────┼─────────────────────────┼────────────
时间           │ 交付前(台架+小海试)    │ 交付后(正式海试)
地点           │ 造船厂/测试基地        │ 目标运营地/海域
参与人         │ 造船厂+船级社+供应商  │ 船公司+造船厂+运维
时长           │ 1-2周                  │ 2-4周
投入成本       │ 较低(本地)            │ 较高(动员船舶)
风险等级       │ 中等(问题可修复)      │ 低(问题已确认)
验收人签字     │ 造船厂工程师          │ 船公司/船级社
通过率(通常)   │ 85%(可能返工)          │ 98%(基本确认)

FAT检查清单：
  □ 启动系统功能(E-STOP正常)
  □ 推进功率输出(无超频/过载)
  □ 冷却系统可靠性(温度控制)
  □ 控制系统响应(指令→执行延迟<500ms)
  □ 数据采集系统(云端上传无延迟)
  □ 故障诊断系统(故障码显示准确)
  □ 多次启停循环(>100次无失效)
  □ 极限工况测试(满载、低温等)

SAT检查清单：
  □ 真实工况运营(不少于50小时)
  □ 实际运行成本验证(油耗/电耗对标)
  □ 故障率观察(观察2-4周内是否有故障)
  □ 船员操作便利性评估
  □ 环保指标确认(零排放确认)
  □ 整体可靠性认可
  □ 性能指标达成(续航/速度/功率)
  □ 远程监控系统验证(云端数据准确)
```

### 20.2.2 FAT详细测试项目(50+项)

**Phase 1: 系统启动与基本功能(8项)**

```
F-01: 冷启动测试
  条件：系统完全关闭≥2小时
  操作：按启动按钮
  预期：≤10秒内推进器达到空载运行
  合格标准：零故障，启动响应时间<10s

F-02: E-STOP功能测试
  操作：按下E-STOP按钮(从驾驶台)
  预期：<3秒推进力降为零
  验证：读取转矩传感器/推进编码器
  合格标准：响应时间≤3s，推进力稳定为0

F-03: 驾驶台显示屏验证
  检查：HMI是否显示全部关键参数
    ├─ 转速(rpm)
    ├─ 推进功率(kW)
    ├─ 水温(°C)
    ├─ 电池SOC(%)
    ├─ 故障代码(若有)
    ├─ 通信状态(在线/离线)
    └─ 运行时间(h)
  合格标准：所有参数显示准确，无缺失

F-04: 控制系统自检
  操作：启动时自动运行自检程序
  预期：所有传感器/执行器绿灯通过
  失败处理：红灯报错，需要维修后方可运行
  合格标准：无自检失败

F-05: CAN总线通信测试
  工具：CAN分析仪
  操作：监听控制器与变频器间的报文交互
  验证：
    ├─ 报文速率正常(250kbps ±5%)
    ├─ 无错误帧
    ├─ 无重复报文
    └─ 无报文丢失
  合格标准：通信正常率≥99.5%

F-06: 云端连接测试
  操作：启动系统，检查云端是否接收到数据
  预期：≤10秒内数据出现在云平台
  验证方式：云平台实时仪表盘显示
  合格标准：首次连接成功率100%

F-07: 温度传感器三通道一致性
  操作：监测电机绕组温度的三个传感器读数
  预期：三个传感器偏差<3°C
  失败处理：更换故障传感器或标定
  合格标准：偏差<3°C

F-08: 编码器信号验证
  操作：旋转推进轴，观察编码器脉冲
  预期：脉冲频率与转速成正比
  计算：pulse_per_rev = 总脉冲 / 转轴圈数
  合格标准：线性度>99%
```

## 20.3 性能验证标准

### 能耗指标验证

```
场景：内河50nm往返测试

参考指标(来自Ch19能耗仿真)：
  设计预测：0.8 kWh/nm
  可接受范围：0.7-0.9 kWh/nm (±10%)

实测计算：
  总能耗：采集整个50nm过程的功率积分
         E_total = ∫P(t)dt (kWh)
  
  持续时间：从出港到回港的总时间(h)
  
  平均功率：E_total / 时间
  
  单位能耗：E_total / 距离(nm)

合格判定：
  □ 单位能耗在0.7-0.9 kWh/nm范围
  □ 效率不低于设计预测的90%
  □ 若不符合，需要诊断原因
    ├─ 冷却系统效率低？(流量不足/温度高)
    ├─ 推进电机效率低？(负载工况不对)
    ├─ 传动损失高？(轴承摩擦增大?)
    └─ 需返工优化设计或工艺

实测示例：
  往程(25nm)：
    累计能耗：20 kWh
    持续时间：2.5h
    平均功率：8 kW
    单位能耗：20/25 = 0.8 kWh/nm ✓ 合格
  
  返程(25nm)：
    累计能耗：15 kWh (顺流，功率需求少)
    持续时间：1.8h
    平均功率：8.3 kW
    单位能耗：15/25 = 0.6 kWh/nm ✓ 合格
  
  日均：(20+15)/50 = 0.7 kWh/nm ✓ 超预期


### 续航能力验证

```
定义：船舶在电池/燃油可用范围内能航行的最大距离

测试方法1：一次性放电法
  初始：电池充满(SOC=100%)
  操作：以巡航速度(8 knot)连续航行至电池低电量告警(SOC=5%)
  记录：航行距离(nm)与时间(h)
  
  计算：续航 = 距离(nm)
        巡航速度 = 距离 / 时间 (knot)
  
  合格标准：续航≥设计预测值×90%

测试方法2：循环模式法(更真实)
  工况：模拟真实运营(不是恒速)
    ├─ 港口启停：0-25%功率，20%时间
    ├─ 加速上升：25-50%功率，10%时间
    ├─ 巡航主行：50-75%功率，60%时间
    └─ 减速靠岸：50-0%功率，10%时间
  
  持续：直至电池耗尽
  
  记录：
    • 每30分钟记录SOC、续航剩余
    • 每小时汇总能耗、速度、风浪状态
  
  输出：续航曲线(时间 vs SOC%)
       并拟合公式用于预测

合格标准：
  □ 续航能力与设计值偏差<10%
  □ 能耗曲线稳定(无异常波动>20%)
  □ 低电量告警准确(偏差<5%)
```

### 故障率与可靠性指标监控

```
关键KPI实时计算：

1. MTBF (Mean Time Between Failures)
   定义：总运行时间 / 故障次数
   
   实时计算示例：
   ┌────────────────────────────────────┐
   │ 船队10艘，各运营6个月              │
   ├────────────────────────────────────┤
   │ 总运营时间：10艘 × 2000h = 20000h │
   │ 总故障次数：25次                   │
   │ MTBF = 20000 / 25 = 800h           │
   │                                    │
   │ 指标评估：                         │
   │ 设计目标：MTBF > 2000h             │
   │ 当前状态：仅800h，✗ 未达成         │
   │ 结论：需要改进，分析根本原因       │
   └────────────────────────────────────┘

2. 故障率曲线(Weibull分布)
   观察：是否处于浴盆曲线哪个阶段
   
   ┌─────┐
   │故障 │        初期故障期    随机故障期    磨损期
   │率  │    \                /           /
   │    │     \            /-\         /
   │    │      \        /-    \     /
   │    │       \    /-        \ /
   │    │        \/-────────────────>时间
   │    │
   └─────┘
   
   判定方法：
   ├─ 若前3个月故障率>30% → 初期故障期(需要烧入/FAT改进)
   ├─ 若6-12个月故障率稳定 → 随机故障期(正常状态)
   └─ 若>12个月故障率上升 → 磨损期(需要维修计划)

3. 可用性(Availability)
   定义：(MTBF) / (MTBF + MTTR)
   
   示例计算：
   MTBF = 2000h
   MTTR = 10h (平均维修时间)
   
   A = 2000 / (2000 + 10) = 0.9950 = 99.50%
   
   指标含义：
   ├─ 99%+ → 优秀(工业标准)
   ├─ 98-99% → 良好
   └─ <98% → 需要改进
```

### 数据分析与改进提案

```
月度数据分析流程：

第1周：数据收集与清洗
  ├─ 导出上月所有原始数据
  ├─ 去除异常值(传感器毛刺等)
  ├─ 填充缺失值(数据传输中断)
  └─ 生成月度统计报告

第2周：故障分析与RCA
  ├─ 汇总故障清单(时间、代码、持续时间)
  ├─ 按故障码分类(Top 5最常见)
  ├─ 对每个Top 5执行RCA
  │  ├─ 故障发生前的参数趋势
  │  ├─ 故障同时发生的其他异常
  │  └─ 根本原因判定
  └─ 生成RCA报告

第3周：可靠性增长评估
  ├─ 绘制MTBF趋势曲线(累计月数)
  ├─ 对标设计目标与改进目标
  ├─ 按船舶/系统统计MTBF差异
  └─ 识别性能最差的船/系统

第4周：改进提案与反馈
  ├─ 针对Top 5故障提出改进建议
  │  ├─ 立即改进(固件补丁、操作规范调整)
  │  ├─ 短期改进(更换部件、设计优化)
  │  └─ 长期改进(新产品升级)
  │
  ├─ 成本-收益评估
  │  ├─ 改进投入(部件成本、工时)
  │  ├─ 预期收益(MTBF增长、故障率降低)
  │  └─ ROI计算
  │
  └─ 提交改进提案至设计/工程部门

输出：
  □ 月度可靠性报告
  □ RCA分析文档
  □ 改进提案单(改进优先级排序)
  □ 技术部门反馈意见
```
