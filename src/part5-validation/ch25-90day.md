# 第24章 90天落地执行路线

> **本章导读**
> 前14章建立了完整的方法论体系——
> 本章解决"如何在90天内让一个新项目跑起来"这个实操问题。
>
> 90天路线图并非要在90天内完成整船研发，
> 而是完成**D0到M3的关键工作**——
> 即需求冻结、架构定型、子系统设计完成、
> 进入HIL/台架试验准备阶段。
>
> **核心原则**：并行推进，重点前置，里程碑门控。

### 15.4.5 新增：架构冻结时间点(M2=Day 60)明确定义

**关键问题**：
"架构冻结"这个概念容易模糊——什么算"冻结"？
是否还允许小改动？什么改动必须停止？

**本节明确**：架构冻结的具体标志与可/不可改动范围

#### 架构冻结的定义

**M2里程碑(Day 60)时，以下物品必须为"冻结状态"**：

✅ **已冻结(不可改动，除非ECR)**：
  · 系统功能框图(Level 1-2)
  · 子系统划分方案(冷却/传动/控制的职责边界)
  · 冗余配置方案(如：双泵/编码器热备)
  · 电压等级(DC 48V / DC 120V等决策)
  · 关键接口ICD v2.0(电气/流体/通信接口)
  · 基础热平衡(散热功率、冷却流量、降额律框架)

⚠️ **受控改动(允许，但需ECR)**：
  · 子系统内部拓扑(不改变对外接口)
  · 传感器类型(如：NTC→PT100，但精度指标不变)
  · 通信报文内容(只要不改频率和实时性要求)
  · 软件模块划分(只要不改硬件I/O接口)

❌ **不允许改动(违反架构原则)**：
  · 从单机推进改为双电机推进
  · 从双泵冗余改为单泵+电加热
  · 改变计算采样周期(>10ms)
  · 改变基础热平衡(K₁/K₂系数)

#### 架构冻结检查清单(M2评审时)

```
☐ N²矩阵完整(所有接口都列出，无遗漏)
☐ ICD v2.0已发布(每个P1/P2接口都有A类文件)
☐ 冷却热平衡初版可运行(Simulink+AMESim联仿)
☐ 传动扭振初版完成(临界转速±10%范围确认)
☐ 控制系统架构图完整(各子模块和通信关系清晰)
☐ FMEA初版≥50条(高RPN项都有设计措施)
☐ 新能源特有方案定稿(热失控防护、低温防护都已决定)
☐ 供应商至少进入技术报价阶段(不再考虑方案变更)
```

#### 为什么Day 60要冻结架构？

D61~D90是"详细设计"阶段，三个子系统团队需要各自深化设计。
如果架构还在变化，下游工作就会返工。

**时间代价**：
- 架构改动 @ D30：影响很小(设计刚启动，< 3天返工)
- 架构改动 @ D60：影响中等(已有初设计，需要7天返工)
- 架构改动 @ D85：影响极大(详设已定，需要15~20天返工)

**M2架构冻结的目的**：
用强制的时间节点，防止"持续微调"导致交期延误

#### 架构冻结的例外(ECR流程)

如果D60~D90期间发现架构缺陷(如：FMEA高RPN新发现)，
可以通过ECR(工程变更请求)申请改动：

ECR流程(参考第11章)：
  1. 提交ECR(描述变更 + 影响范围估算)
  2. CCB评审(成本/工期/风险评估)
  3. 若批准：可改动 + 更新ICD v2.1 + 关联VCRM回退
  4. 若拒绝：维持现状或推迟至下一产品版本

**关键原则**：
ECR数量若>3个，说明架构有根本缺陷，需要反思第一阶段工作质量

## 15.7 常见90天踩坑记录（经验警告）

```
以下是新能源电推项目在D0~D90阶段最常见的10个失误：

  ✗ 坑1：需求还没冻结就开始选型
    症状：Day 15就在问"哪个品牌电机好"，需求SRS一行没写
    后果：需求变更→设计推倒重来→供应商索赔
    预防：M1（Day 30）需求冻结前，供应商只发RFI（信息询问），不发RFQ（报价询价）

  ✗ 坑2：船型定位模糊（内河vs海船摇摆不定）
    症状：D30了还在说"这条船可能跑近海"
    后果：规范适用性不确定→设计余量无法确定→CCS审图反复
    预防：D0当天强制完成第18章决策树，书面确认，各方签字

  ✗ 坑3：FMEA流于形式（照抄模板，不结合实际）
    症状：FMEA里全是"失效后果：设备损坏"，无定量分析
    后果：真实高风险失效模式（螺旋桨出水飞车/热失控蔓延）漏掉
    预防：FMEA评审会邀请船东运维方参加，逼出真实场景

  ✗ 坑4：降额律只有K₁（没有K₂）
    症状：设计文件里只有"海水温度过高时降额"，换热器污堵工况没考虑
    后果：换热器3年后污堵50%，绕组烧毁，船东索赔
    预防：M2计算书强制包含K₂（污堵降额律），与K₁联合仿真验证

  ✗ 坑5：HIL平台D60才开始搭建
    症状：D60才订HIL硬件，D90时平台刚装好，一个场景没跑
    后果：台架试验期间发现控制逻辑错误，现场改代码，延误交期
    预防：HIL平台硬件D30即采购，D50前完成场景01~05

  ✗ 坑6：电池热失控传播测试报告"等供应商提供"
    症状：D80时才问供应商要热失控报告，供应商说"没做过，需要3个月"
    后果：CCS审图资料不完整，无法入级，交期延误
    预防：D0供应商选型条件中明确：热失控传播测试报告是入围必要条件

  ✗ 坑7：轴承电腐蚀问题没人负责
    症状：电机工程师说"这是变频器问题"，变频器工程师说"这是电机问题"
    后果：无人设计防护措施，轴承1年内发出异响，电腐蚀麻点确认
    预防：D0指定系统工程师负责跨子系统问题（轴承电腐蚀、谐波等）

  ✗ 坑8：充电谐波THD测量在航行试验才第一次做
    症状：SAT充电谐波测试THD=12%（超标两倍），整改需新增APF
    后果：交船前紧急加装设备，改图，CCS补充审图，延期2个月
    预防：系泊试验SA09即进行谐波测量，问题早发现早整改

  ✗ 坑9：低温充电限制策略未经-10°C台架验证
    症状：冬季靠港充电，虽然BMS有保护，但PTC加热片功率不足
          电池从-15°C加热到0°C需要2.5h，船员等不及强行充电
    后果：充电接触器因船员绕过BMS手动操作而损坏（更严重风险）
    预防：M4 FAT阶段在环境箱内验证低温充电全流程（包括预热时间）

  ✗ 坑10：VCRM只在审查前更新
    症状：平时不维护VCRM，M5前两周突击关闭95%条目
          部分条目找不到验证证据（试验早做完了但报告没编号）
    后果：CCS审查时发现证据链断裂，需要补做试验
    预防：VCRM每周更新一次（责任人制度），每个关闭条目当天录入证据编号
```
